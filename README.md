# RGBD 스터드 타이어 홀 검사 (RealSense D435)

3D 프린트한 스터드/스파이크 타이어 표면 모형(검은색 PLA)에서, 실제로 스터드 핀(압정)을
박기 전에 설계된 홀이 제대로 뚫려 있는지 RealSense D435로 검사하는 프로젝트.

## 왜 Depth 기반인가

출력물이 무광 검은색 플라스틱이라 RGB만으로는 홀과 주변 표면의 대비가 거의 없다. 그래서
depth 채널로 "주변 표면보다 국소적으로 더 멀리(더 깊이) 측정되는 영역"을 찾는다. 다만 RGB도
Hough Circle로 원형 후보를 찾는 데 보조적으로 사용한다(아래 알고리즘 참고) - depth만으로
후보를 찾으면 트레드 문양 때문에 baseline이 오염되는 문제가 있어서, 후보 검출과 depth 검증
단계를 분리했다.

## 파일 구성

| 파일 | 역할 |
|---|---|
| `model.py` | 메인 실행 스크립트. D435 캡처 -> 정렬(align) -> 홀 탐지 -> 화면 표시 -> `s`로 결과 저장 |
| `hole_detector.py` | 핵심 탐지 알고리즘 (`HoleDetector`, `HoleDetectorConfig`) |
| `calibration.py` | 카메라 좌표 -> 로봇 베이스 좌표 변환 (`hand_eye_calibration.json` 로드) |
| `hand_eye_calibration.example.json` | 핸드아이 캘리브레이션 결과 파일 형식 예시 + 동차좌표/캘리브레이션 절차 설명 |
| `distance_calibration.py` | 거리별 depth 노이즈 실측 도구 - 클릭한 지점을 여러 거리에서 측정해 노이즈-거리 그래프를 그려줌 (최적 작업 거리 찾기용) |
| `requirements.txt` | 의존 패키지 목록 |

## 설치 및 실행

```bash
pip install -r requirements.txt
python model.py
```

조작: `q` 종료, `s` 현재 프레임 홀 좌표를 `holes_<timestamp>.json`으로 저장, `c` 30프레임
평균 캡처로 재탐지, `r` 실시간 트래커 초기화. 아직 안 뚫린 홀(빨간 원)과 이미 스터드가
박힌 자리(주황 원, 정상/틀어짐/덜박힘)를 **매 프레임 둘 다 자동으로** 찾아 같이 표시한다
(아래 "스터드 삽입 후 상태 검사" 참고) - 좌표를 따로 기억해둘 필요가 없다. 미리보기 창
클릭 시 그 지점의 실제 depth(m/mm)를 콘솔에 출력하고, 진단은 항상 스터드 상태 기준으로
나온다.

## 탐지 알고리즘 개요

(v1 morphological-opening 방식에서, 별도 프로젝트의 검증된 접근법을 참고해 v2로 교체함 -
아래 "참고한 선행 코드" 참고)

1. **물체 분리** (`segment_object`): 설정된 거리 범위(`min/max_object_distance_m`) 안에서
   가장 큰 연결 성분을 타이어 출력물로 간주하고 배경(책상, 벽, 손 등)을 제외한다.
2. **원형 후보 검출** (Hough Circle, RGB + depth 양쪽): CLAHE로 보정한 흑백 RGB와, depth의
   국소 굴곡을 밝기로 표현한 "depression 이미지" 양쪽에서 `cv2.HoughCircles`로 원형 후보를
   찾아 합친다. depth로 baseline(=매끈한 국소 표면, 가우시안 블러로 추정)을 만드는 건
   이때뿐이고, 이미지 전체가 아니라 후보 검출용으로만 쓰여서 V자 트레드 띠가 전체 판정을
   오염시키지 않는다.
3. **후보별 링(ring) 검증** (`_circle_depth_evidence`): 각 후보를 감싸는 고리 모양 영역의
   depth만 참조값으로 써서, 후보 중심이 그 링보다 `min~max_hole_depth_mm` 범위만큼 더
   먼지(=파여있는지) 확인한다. 국소 커널이 옆 구조물까지 걸치는 v1의 baseline 오염 문제가
   구조적으로 없다.
4. **8방향 섹터 검사**: 후보 주변을 8방향으로 나눠 몇 방향에서 깊은지 센다
   (`required_deep_sectors`). 진짜 둥근 홀은 사방에서 깊고, 트레드 홈(V자 띠)은 홈이
   지나가는 한두 방향에서만 깊어서 이 차이로 확실히 구분된다.
5. **그루브 연결성 검사** (`_connects_to_outside_depression`): 후보의 깊은 영역이 반경
   1.75배 밖까지 이어지면 고립된 홀이 아니라 그루브의 일부로 보고 제외한다.
6. **3D 좌표 변환**: 살아남은 후보마다 링의 depth로 실지름(mm)을 재확인하고, 중심 픽셀 ->
   카메라 3D 좌표(`rs2_deproject_pixel_to_point`)까지 계산한다.
7. **로봇 좌표 변환** (`calibration.py`): 4x4 동차 변환 행렬로 카메라 좌표를 로봇 베이스
   좌표로 옮긴다. 캘리브레이션 파일이 없으면 identity로 대체하고 경고를 띄운다.

### 참고한 선행 코드

`~/Desktop/카메라코드_1 1/카메라코드/depth_v1.py` - 실제 고무 타이어 대상으로 같은 문제
(스터드 홀 vs 트레드 홈 구분)를 이미 다뤘던 코드. Hough+링+8섹터+그루브 연결성 검사 아이디어를
그대로 가져왔고, `HoleTracker`(프레임 간 추적/안정화)는 2단계(실시간) 작업 때 도입했다.

### 추가 후보 검출/검증 기법 (실측 검증 전까지 기본은 꺼둠)

Hough Circle(RGB+depth) 단독으로 부족한 경우를 보완하려고 아래 세 기법을 추가했다.
전부 `HoleDetectorConfig`의 `enable_*` 플래그로 켜고 끌 수 있고, 실측 비교 결과 아직은
Hough 단독이 더 깔끔해서 기본값은 모두 꺼져 있다(오탐지가 늘어나는 게 확인됨 - 특히
가장자리 완화와 같이 켰을 때 하단 영역에서 오탐지 급증).

| 기법 | 설정 | 아이디어 |
|---|---|---|
| LoG(Laplacian of Gaussian) 후보 검출 | `enable_log_candidates` | 테두리가 아니라 "정해진 크기의 둥근 덩어리 자체"를 직접 찾음 - Hough가 놓치는 흐릿한 테두리 후보 보완 |
| 템플릿 매칭 2차 패스 | `enable_template_matching` | 이번 프레임에서 이미 확정된 홀 하나를 템플릿으로 잘라 화면 전체에서 닮은 자리 추가 검색 |
| RGB 그림자/하이라이트 검사 | `enable_shadow_highlight_check` | 방향성 조명 아래 진짜 오목한 홀에 생기는 "한쪽 그늘/반대쪽 밝음" 패턴을 8방향 밝기에 코사인 적합(R²)으로 점수화 |

물체 가장자리 근처에서 검증 기준을 완화하는 로직(`is_edge`/`edge_ratio`)도 한때 있었지만,
하단 영역 오탐지의 원인으로 지목되어 **완전히 제거**했다 - 지금은 물체 어디서든 동일하게
엄격한 기준(`_evaluate_candidate`)을 적용한다.

## 스터드(못) 삽입 후 상태 검사 (신규 기능, 실측 검증 전)

로봇이 스터드를 박은 뒤, 그 자리가 제대로 박혔는지 검사하는 별도 기능. 삽입 전 홀 탐지와는
목적이 달라서(빈 홀 vs 이미 채워진 스터드) 별도 함수들(`HoleDetector.detect_stud_states`,
`classify_stud_state`)로 분리했다.

- **위치 탐지는 RGB 은색 픽셀만으로** (`_detect_stud_head_candidates`): 스터드가
  박히면 더 이상 "홀"처럼 안 보여서 depth Hough 파이프라인으로는 재탐지가 안 된다.
  처음엔 삽입 전 홀 좌표를 그대로 고정해서 재사용했었는데, 카메라/물체가 삽입 도중
  움직이면 좌표가 어긋나는 문제가 있었다. 그 다음엔 "빈 홀과 같은 depth 굴곡(Hough)
  후보 중 안에 은색이 있는 것만" 방식으로 덜 박힌 케이스를 잡으려 했는데, **실측
  결과 이 방식 자체가 구조적으로 안 맞았다**: 덜 박힌 스터드는 일부는 이미 표면
  높이까지 들어갔고 일부만 아직 오목해서, 빈 홀처럼 깔끔한 원형 굴곡이 아니라
  Hough가 애초에 원 후보 자체를 못 찾는 경우가 많았다 - depth 신호에 의존하는 한
  못 고치는 한계였다. 그래서 depth 후보를 아예 거치지 않고 RGB만으로 두 크기
  등급을 찾는다:
  1. **완전 블롭** (면적 ≥ 정상 크기 기준): 온전히 다 보이는 머리 - 완전히 박혀
     표면과 거의 같은 높이(flush)라 depth 굴곡이 거의 없는 스터드를 잡는다.
  2. **조각** (`stud_partial_min_silver_area_mm2` ~ 정상 크기 기준 사이): 구멍
     벽에 가려(+그늘져) 일부만 보이는 경우. 조각 자체의 넓이는 실제 크기를
     반영 못 하므로 설계상 알고 있는 정상 크기를 그대로 쓴다. 이보다 작으면
     노이즈(반사광 한두 픽셀)로 보고 버린다 - "은색이 하나라도 있으면" 기준을
     썼다가 노이즈가 스터드로 오탐지되는 문제가 실측으로 확인돼서 최소 면적
     기준을 뒀다.
  두 등급 다 R/G/B 채널 편차(chroma, `stud_head_max_chroma`)가 작고 밝은
  (`stud_head_min_brightness`) 픽셀만 후보로 본다 - HSV 채도 대신 R/G/B 편차를
  직접 보는 이유: 카드보드 박스/나무 바닥처럼 밝지만 갈색/베이지 색조가 있는
  배경이 물체와 비슷한 거리에 걸쳐 object_mask 안에 들어오면 HSV 채도만으로는
  안 걸러지는 게 실측으로 확인됐다. 좌표를 기억해둘 필요가 없어 카메라/물체가
  움직여도 안전하다.
- **판정 기준** (위치를 찾은 뒤 `classify_stud_state`가 그 자리의 depth로 판정):
  - 안착 정도(`seating_offset_mm`, 링 검증과 동일한 recess_mm 재사용): 양수=주변 표면보다
    안쪽/멀다(아직 설계 깊이만큼 다 안 들어가 오목한 굴곡이 남음), 음수=튀어나옴/가깝다
    (표면 밖으로 나옴). "정상(flush)"은 이 값이 0 근처(`stud_seating_tolerance_mm`
    이내)일 때뿐이고, 양쪽 어느 방향으로 벗어나도 "덜 박힘(under_inserted)"이다.
    recess가 설계 홀 깊이 상한(`stud_partial_max_depth_mm=3.0`)보다 크면 신뢰할 수
    없는 값(예: 빈 홀을 스터드로 착각)으로 보고 "원인불명"으로 처리한다.
  - 기울어짐: 스터드 머리 표면 자체를 8방향으로 나눠(`_face_sector_depths` - 기존 링
    섹터와 달리 머리 표면 안쪽을 봄) depth에 코사인 하나를 맞춘 진폭/적합도(R²)가 둘 다
    기준(`stud_tilt_amplitude_mm`, `stud_tilt_r2`)을 넘으면 "틀어짐(tilted)". 다만 카메라가
    물체 중앙 위에 있으면 화면 가장자리 쪽 스터드는 순전히 시야각(원근) 때문에 실제로
    안 기울었어도 미세한 depth 기울기가 생길 수 있어, 이 코사인 적합이 그걸 오탐할
    위험이 아직 실측으로 확인/보정되지 않았다.
- **사용법**: `model.py`가 매 프레임 아직 안 뚫린 홀(빨간 원)과 이미 스터드가 박힌 자리
  (주황 원)를 동시에 찾아 표시한다 - 상태는 색이 아니라 라벨 텍스트(정상/틀어짐/덜박힘/
  원인불명)로 구분. 미리보기 클릭 진단도 항상 스터드 상태(`상태/seating_offset/
  tilt_amplitude/tilt_r2`) 기준으로 나온다.
- **검증 상태**: 위치 탐지(RGB 두 등급)와 판정(depth) 각각 합성 데이터로는 확인했지만
  (정상/틀어짐/덜박힘/무효 4가지 + 노이즈 강건성 + 부분 노출된 작은 은색 조각 케이스 +
  반사광 노이즈 산발 배치 오탐지 방지), 임계값 4개(`stud_seating_tolerance_mm=1.0`,
  `stud_tilt_amplitude_mm=1.2`, `stud_tilt_r2=0.6`, `stud_partial_max_depth_mm=3.0`)와
  RGB 임계값 3개(`stud_head_min_brightness=140`, `stud_head_max_chroma=25`,
  `stud_partial_min_silver_area_mm2=1.0`) 전부 기본값이라 **실측 검증 및 튜닝이
  필요**하다. 실측 3곳(정상/틀어짐/덜박힘) 중 2곳(정상, 틀어짐)은 판정이 일치했으나,
  "정상" 판정의 seating_offset이 허용 오차 경계선(-1.00mm, 기준 1.0mm)에 딱 걸쳐 있어
  여유가 거의 없었다. "덜박힘"은 여러 차례 실측에서 계속 탐지 자체가 안 됐던
  케이스라(처음엔 RGB 블롭 단독, 그다음엔 depth+은색 병합 - 둘 다 실패) 지금은
  depth 후보를 아예 안 거치고 RGB 두 크기 등급으로 다시 접근했고, 아직 실측
  재검증 전이다.

## 실제 부품 스펙

- 홀(스터드 핀 머리 입구): 지름 11mm, 깊이 3mm — **카메라가 찾아야 하는 대상**
- 핀 끝부분: 지름 2mm, 깊이 7mm — 입구 안쪽 깊은 곳이라 D435 해상도 밖, 비전 탐지 대상 아님
  (로봇이 11mm 입구 중심만 정확히 맞추면 됨)
- 타이어 표면 문양이 V자 형태라, 홀뿐 아니라 트레드 문양 자체에도 depth 굴곡이 있다 ->
  8방향 섹터 검사 + 그루브 연결성 검사가 "진짜 홀 vs 트레드 문양"을 가르는 핵심 로직

## 알려진 근본 문제 (실측으로 확인됨)

파라미터 튜닝만으로는 못 고치는 것과, 알고리즘을 바꾸면 고칠 수 있는 것을 구분해둔다.

| 문제 | 원인 | 상태 |
|---|---|---|
| 신호가 약함 | 설계 깊이 3mm인데 실측은 1~2mm뿐. D435 노이즈 수준과 비슷해서 구분이 어려움 | ⚠️ 하드웨어 한계 (다중 프레임 평균 + IR 프로젝터 최대 출력으로 완화만 가능) |
| depth 1mm 단위 양자화 | 센서가 정수 mm로만 값을 냄 | ⚠️ 하드웨어 한계 |
| V자 트레드 띠가 홀 바로 옆 baseline을 오염시킴 | 리브 폭이 좁아서 baseline 추정 커널이 옆 띠까지 걸침 | ✅ 해결 - Hough+링+8섹터 방식으로 알고리즘 교체 (합성 데이터로 검증 완료, 실측 검증은 Sprint 2 진행 중) |
| 임계값 하나로 "진짜 홀 vs 띠"가 안 갈림 | 기존 방식은 전역 residual 임계값 하나뿐이었음 | ✅ 해결 - 8방향 섹터 검사 + 그루브 연결성 검사로 형태 기반 구분 추가 |
| 오래 켜두면 화면이 점점 느려짐/과거를 보여줌 | 프레임 처리(양방향 필터+Hough 2회+8섹터)가 카메라 프레임 주기(33ms)보다 오래 걸려 librealsense 내부 큐에 프레임이 쌓임 | ✅ 해결 - `poll_for_frames`로 매 루프 가장 최신 프레임만 처리, 밀린 프레임은 버림 |

- **핸드아이 캘리브레이션 미완료**: `hand_eye_calibration.json`이 아직 없어 로봇 좌표
  변환이 identity(카메라 좌표 그대로)로 동작 중. `hand_eye_calibration.example.json`에
  절차 설명 있음.
- **미해결: 특정 3개 지점에서 실측 결과가 애매함** - 클릭 진단 결과 recess가 약하거나
  (`invalid_fraction` 높음, 프레임마다 통과/탈락이 왔다갔다 할 수 있는 경계선) 부호가
  반대로 나오는(주변보다 튀어나온 것처럼 측정됨) 지점이 있었음. 스터드 검사 모드(`i`)로
  재확인 필요 - 진행 중.
- **스터드 삽입 후 상태 검사 임계값 미검증**: 위 "스터드 삽입 후 상태 검사" 절 참고,
  합성 데이터로만 확인됨.

## 개발 로드맵 (애자일 스프린트)

**최종 목표**: 실시간으로 홀 좌표를 탐지하며 정확도 90% 이상.
**전략**: 조건을 고정한 정지 이미지에서 먼저 정확도 100%를 확정한 뒤(1단계), 카메라/물체가
움직이거나 조건이 바뀌는 실시간 상황으로 확장한다(2단계). 각 스프린트는 이전 스프린트의
완료 기준(DoD)을 통과해야 다음으로 넘어간다.

### 1단계 — 정지 이미지 100% 정확도

**Sprint 1. 기준선 측정 (계측 없이는 개선도 없다)**
- 목표: "정확도"를 숫자로 정의하고 현재 값을 안다
- 작업: 실제 홀 총 개수(도면 기준) ground truth 확정 → `c`(평균 캡처)로 여러 번 반복 촬영
  → 홀마다 정탐/오탐/미탐 기록
- 지표: 정탐율 = 정탐 홀 수 / 전체 진짜 홀 수, 오탐지 수 = 가짜로 잡힌 개수
- DoD: 현재 수치가 문서(이 표 아래)에 기록됨

**Sprint 2. baseline 오염 문제 해결** — 알고리즘 교체 완료(합성 데이터 검증), 실측 검증 남음
- 목표: V자 띠가 옆 홀의 baseline을 망가뜨리는 근본 원인 제거
- 작업: Hough Circle(RGB+depth) + 후보별 링 검증 + 8방향 섹터 검사 + 그루브 연결성 검사로
  교체 (`hole_detector.py`, 선행 코드 `depth_v1.py` 참고)
- DoD: Sprint 1 대비 정탐율 상승 + 오탐지 감소가 반복 캡처로 재현됨 - **다음 할 일: 실제
  캡처로 재측정**

**Sprint 3. 파라미터 최종 확정 + 100% 검증**
- 목표: 남은 임계값(`min/max_hole_depth_mm`, `hough_param1/2`, `required_deep_sectors`,
  `connection_depth_mm`) 확정
- 작업: 여러 회 반복 캡처로 grid search, 안정적인 값 확정
- DoD: **동일 조건에서 N회(예: 10회) 반복 캡처 시 100% 정탐 + 오탐지 0**이 재현되어야 통과

### 2단계 — 실시간 90%+

**Sprint 4. 단일 프레임(실시간) 성능 확보**
- 목표: 30프레임 평균 캡처 없이 매 프레임(또는 짧은 슬라이딩 윈도우) 처리로도 정확도 확보
- 작업: 단일 프레임 vs 평균 캡처 정확도 비교, 필요시 최근 N프레임 슬라이딩 윈도우로 절충
- DoD: 실시간 프레임레이트를 유지하며 정탐율 90%+ 재현

**Sprint 5. 움직임/조건 변화 대응**
- 목표: 카메라나 물체가 약간 움직이거나 각도/거리가 바뀌어도 안정적으로 탐지
- 작업: 여러 각도·거리 조건 데이터 수집, 거리 범위 자동 추정(고정값 대신 매 프레임 추정),
  흔들림 대응 테스트
- DoD: 정해진 변화 시나리오(예: ±10도 각도, ±5cm 거리)에서 정탐율 90%+ 유지

**Sprint 6. 로봇 연동 통합**
- 목표: 실시간 탐지 좌표를 실제 로봇 좌표로 변환해 전달, end-to-end 동작 확인
- 작업: 핸드아이 캘리브레이션 실측, 로봇 통신 연결, 카메라→탐지→좌표변환→로봇 이동
  전 과정 테스트
- DoD: 로봇이 실시간 탐지 좌표로 스터드 삽입을 시도해 90%+ 성공률 달성
